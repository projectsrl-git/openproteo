Fix csvsql: inline CSVREAD path/options as SQL literals

csvsql steps failed at staging with H2 error 90012
  Parameter "fileName" is not set; SQL statement:
  CREATE TABLE <t> AS SELECT * FROM CSVREAD(?, NULL, ?)
even with a correct, existing input path.

Cause: for CREATE TABLE ... AS SELECT, H2 derives the table columns at
statement-PREPARE time and (with a NULL column list) must read the CSV header
then -- before JDBC bound parameters are applied -- so the bound CSVREAD file
name is seen as unset. Known H2 limitation, not a path issue.

Fix: inline the file name and options as escaped SQL string literals (single
quotes doubled) via a new sqlLit() helper, instead of bound '?' parameters. The
table name is already regex-validated and the literals are escaped, so this is
not an injection vector.

- engine/InternalSteps.runCsvSql: Statement + CSVREAD('path', NULL, 'opts').
- web/ApiController csvsql preview: same.
- h2/H2Probe.java: had the same bound-parameter bug (would fail the link test on
  a healthy H2); switched to the inlined-literal form to match the executor.

Compiles (96 classes). Re-run the csvsql step and h2/H2Probe on UBS to confirm
(live H2 is not runnable in the build sandbox).
