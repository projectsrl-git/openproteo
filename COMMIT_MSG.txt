Speed up csvsql: in-memory H2 engine + per-input indexes

csvsql was correct but unusably slow on ~1M-row inputs with joins/filters: every
input was materialised into an on-disk H2 (MVStore) database and the all-VARCHAR
tables had no indexes, so each join/filter was a full scan.

- Engine selection (step param 'engine', default orchestrator.csvsql-engine =
  auto): auto uses an in-memory H2 below orchestrator.csvsql-mem-max-mb (default
  512 MB of total input) and on-disk above it; 'mem' forces in-memory (fastest);
  'file' forces on-disk. The chosen engine + input size are logged.
- Per-input indexes: <input ... index="NDG,CODCLI"> builds CREATE INDEX on the
  listed columns after staging (regex-validated; unknown columns warn, not fatal)
  -- the big win for complex queries.
- H2 tuning: LOCK_MODE=0 and CACHE_SIZE=262144 on the connection.
- In-memory DBs use a unique name + DB_CLOSE_DELAY=0 (freed on close); on-disk
  cleanup made null-safe.
- Designer: per-input "Index cols" field + an "Engine" selector on the output row
  (auto/in-memory/on-disk). Preview already used capped in-memory H2.

Guidance: for big joins keep Engine=auto and index the join keys; if mem hits
OutOfMemory set Engine=file (still indexed) or raise -Xmx.

Compiles (96 classes). Round-trip test for input delimiter+index passes; live H2
timing is UBS-side.
