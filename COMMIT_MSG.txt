Fix diff CSV_KEY performance: materialise ag/bg instead of CTEs

CSV_KEY reconciliation was extremely slow at scale. runDiffKey ran the whole
comparison as one WITH ag/bg ... SELECT ... UNION ALL ... statement that
references ag and bg twice each (LEFT JOIN + anti-join). H2 does not materialise
non-recursive CTEs, so each per-side GROUP BY ... COUNT(DISTINCT) ran about four
times and the two grouped subqueries were joined without an index; cost grew
super-linearly (about 44 s on 20k x 20k rows).

Fix: materialise the two grouped sides into indexed temporary tables, then join:
- CREATE LOCAL TEMPORARY TABLE ag AS <agSelect> / bg AS <bgSelect> (each GROUP BY
  runs once),
- CREATE INDEX ix_ag ON ag(k) / ix_bg ON bg(k) (the key join is an index lookup),
- the final query drops the WITH and selects from the ag/bg tables.
Temp tables are session-scoped on the per-run in-memory H2 connection, so they
need no cleanup and cannot collide.

Same output (the 5-key sample still reports DIFFERENCES(4) with identical
categories). 20k x 20k now runs in about 2 s through the real runDiffKey, roughly
20-30x faster and roughly linear.

Verify: the patched runDiffKey was run against real H2 2.1.214 -- correctness
sample unchanged; timing 44 s -> ~2 s. Full Maven build to be confirmed on deploy.
