Adaptive, resource-gated parallel run scheduler (G3)

Lets different feeds run in parallel when resources allow, instead of strict
single-thread FIFO. Same feed stays serialised; default preserves old behaviour.

- Config: orchestrator.max-parallel-runs (default 1 = legacy sequential),
  orchestrator.run-admission-headroom-mb (default 256), orchestrator
  .scheduler-tick-sec (default 20).
- Engine: single-thread executor replaced by a cached daemon pool bounded by an
  admission step. start() and gate-resume now enqueue(runId, task); a synchronized
  schedule() admits while active < max-parallel AND (active==0 OR free heap >=
  headroom), and each finishing/pausing task re-runs schedule() so a freed slot
  admits the next. A wf-scheduler tick re-checks resources every tick so runs
  deferred for memory get admitted when heap frees. Per-feed lock unchanged
  (no two runs of one feed); test runs use their own executor.
- Operations Resources panel shows "Parallelism (admitted / max)" + runs waiting
  for a slot (new fields on /api/overview/resources).

Default max-parallel-runs=1 reproduces previous behaviour exactly. Admission
algorithm simulated standalone (peak concurrency == max for {1,3,5}, all tasks
completed, no deadlock). Compiles (99 classes).
