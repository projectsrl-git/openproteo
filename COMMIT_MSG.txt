csvsql: enforce query timeout + ANALYZE (runaway queries)

A 3-level self-join of a 1.5M-row table with OR conditions in the JOIN
predicates ran for hours. Two issues:

1) Tool bug: internal steps (csvsql/xlsx2csv) ignored the step timeout entirely
   (only external script steps got it), so a runaway query had nothing to stop
   it.
2) The query itself is pathological: an OR in a join condition cannot use an
   index/hash join and degrades to a nested loop (rows x rows ~ 2.25e12 for the
   first join, multiplied again by the second), which no engine tuning can fix.

Changes (tool side):
- csvsql applies Statement.setQueryTimeout to the staging, index and query
  statements, from the step's TIMEOUT SEC (else orchestrator.default-step-
  timeout-sec = 1800s). A query over the limit is aborted by H2 and the step is
  flagged timedOut with a clear message, instead of hanging for hours.
- best-effort ANALYZE after staging+indexing so H2 has stats for join order /
  index use.

Guidance (USAGE): avoid OR in JOINs; rewrite each OR-join as a UNION of separate
equi-joins. csvsql targets filter/aggregate/equi-join workloads, not multi-hop
graph expansion over millions of rows.

Compiles (96 classes). Live H2 timing is UBS-side.
