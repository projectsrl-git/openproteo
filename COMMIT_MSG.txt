Operations: live RUNNING flag; Stop: force-stop on repeat + idempotent finish

Operations kept showing a feed RUNNING after its run ended, and a stuck AS400 SQL
extraction needed more than one Stop.

- /api/overview/feeds: the running flag was cached with the heavy last-status disk
  reads (10s TTL). Now last-status stays cached but `running` and `bucket` are
  recomputed every call from in-memory engine state (activeRunId), so a feed drops
  out of RUNNING the instant its run leaves activeRuns.
- engine.stop(): pressing Stop again while still active force-finalises the run
  ABORTED and abandons the (genuinely stuck) worker, so the UI/Operations never
  stay stuck on RUNNING. Audited RUN_FORCE_STOPPED. First Stop is still the graceful
  cancel + DB-connection-close; second is the escape hatch.
- finish() made idempotent + thread-safe (synchronized check-and-claim) so the
  abandoned worker can't re-finalise after a force-stop.
- run.html: generic Stop confirm text; button relabels to "Force stop" after the
  first press.

Compiles (101 classes). Complements the previously shipped sql/DB2 connection-close
aborter (deploy that too).
