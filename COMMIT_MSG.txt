Per-part row counts (csvRowCounts) + optional parallel loop list

Expose per-part row counts from the SQL and SPLIT executors, and let a LOOP
iterate a second list in lockstep, so a workflow can read each part's row count
without re-reading the files. Fully backward compatible.

Executors (new output variable, additive):
- ds/CsvWriter: track data rows per part (partRows), aligned with files.
- ds/SqlSupport.ExportResult: carry partRows from the writer.
- engine/InternalSteps: sql, csvsql and split now also emit csvRowCounts -- the
  per-part data-row counts, joined with the same delimiter and in the same order
  as csvFiles (so sum(csvRowCounts) == rowCount by construction). New joinCounts
  helper; split tracks per-part counts in its own loop.

LOOP parallel list (opt-in, defaults preserve current behaviour):
- model/def/LoopDef: optional over2 (parallel list) and itemVar2 (default
  "item2", only set when over2 is present).
- engine/WorkflowEngine: when over2 is set, split it with the loop delimiter and
  expose itemVar2 per iteration, indexed in lockstep with the main list (empty
  when shorter); the parallel list state is cleaned up at loop end. No over2 ->
  identical to before.
- parser reads over2/itemVar2; writer emits them only when over2 is set, so
  existing workflow files stay byte-identical.
- web/dto + toDto carry the two fields; designer adds an "Over 2" and
  "Item var 2" field and seeds them on new loops.

Compiles (104 classes); designer passes node --check.
