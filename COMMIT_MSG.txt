Fix: never pass orchestrator-internal step params to the external script

deleteOnSuccess, deleteOnSuccessType and outputData.* are engine configuration
stored as step params, but StepExecutor was forwarding every param to the
script: PowerShell as -name 'value' (so -deleteOnSuccessType 'file' triggered
"A parameter cannot be found that matches parameter name 'deleteOnSuccessType'"
/ NamedParameterNotFound), CMD and JAR as positional values (shifting the real
arguments).

- engine/StepExecutor.buildCommand: filter out reserved params before building
  the command line, for all kinds (POWERSHELL, CMD, JAR). New isReservedParam()
  helper (deleteOnSuccess, deleteOnSuccessType, outputData.*).
- templates/designer.html: hide those reserved params from the visible
  PARAMETERS list (real indices preserved). They are still managed by their own
  controls ("On success delete", "Output data") and still written to the XML.

No feed change needed: existing workflows run correctly after redeploy; the
internal params remain in the XML and keep configuring the engine.

Compiles (104 classes); designer passes node --check.
