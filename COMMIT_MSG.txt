Normalise CR/LF inside values at the JDBC extraction, with conservative defaults

Reassembling a record that was split across physical lines needs a quote-parity heuristic in the
dequote executor. The right place is the extraction itself: there the column count comes from
ResultSetMetaData, so the shape of a record is known exactly, and a value containing CR/LF is the
only thing that can break "one record = one physical line". SqlSupport now normalises the value
while writing it, which is a single funnel every extracted cell passes through.

- nlReplacement(mode) maps `space` to a single space, `strip` to nothing and anything else to
  keeping the value as-is.
- exportResultSet and exportCsv gained an overload taking the mode; the pre-existing signatures
  delegate with "keep", so no caller changes behaviour.
- ExportResult.newlinesSanitized counts the cells that were touched. The sql (JDBC extraction) and
  csvsql steps read the step parameter `newlinesInValues` and publish ${newlinesSanitized}; the sql
  step also logs a line when it is greater than zero. The designer exposes the choice as
  "Line breaks inside extracted values".

Defaults are deliberately conservative, because production feeds are already configured the old
way and must not change on deploy:

- extraction `newlinesInValues` defaults to **keep**: values are written exactly as the database
  returns them, byte-for-byte as before;
- dequote `embeddedNewlines` now defaults to **keep**, i.e. records are not joined. The previous
  patch defaulted to `space`, which silently changed the output of every existing dequote step;
- dequote blank-line dropping now sits behind `dropBlankLines`, default **no**. The previous patch
  dropped blank lines unconditionally, with no way to opt out.

Both are opt-in per step from the designer, or massively from the Variables page and the matrix.
