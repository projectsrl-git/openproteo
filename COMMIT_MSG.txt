Duplicate workflow copies uploaded files; internal viewer aggregate respects range filters

#1 Duplicating a workflow ("Duplicate as new") recreated only the definition; the uploaded
files (dataschema/displayschema/scripts) were not copied, leaving the new feed's directory
incomplete. Added AssetStore.copyFeedAssets(from,to) (copies each uploaded file + its
metadata, writing metadata once) and a new endpoint
POST /api/workflows/{feedId}/copy-assets-from/{sourceFeedId}. designer.html now remembers
the source feed on duplicate (window.DUP_FROM) and, after a successful save (both the form
save() and saveXmlDirect()), copies the files into the new feed and remounts/reloads, so
the duplicate is a faithful copy.

#4 (internal viewer) The aggregate ignored the per-column range filters: csvAgg only took
group/distinct/q, so CsvService.aggregate never saw them. CsvService.aggregate now has an
overload taking List<Filter> (cache key includes them; the scan applies matchesFilters like
page() does); the old 4-arg form delegates with null. csvAgg and both @GetMapping wrappers
accept fc/ff/ft and build filters exactly like csvPage. viewer.js buildAgg receives a
getRanges callback and appends &fc/&ff/&ft to the aggregate URL, so the Aggregate tab
reflects the active range filters.

Docs: .claude/2026-07-21-duplicate-assets-and-internal-agg-range.md and CLAUDE.md.

Thymeleaf); CsvService/ApiController/AssetStore are brace/paren balanced and mirror the
existing csvPage / AssetStore code. Java not compiled in the chat sandbox -- run the full
Maven build on deploy.
