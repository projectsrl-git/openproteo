Step: optional "on success delete file/directory" cleanup

Each step can declare a file or directory to remove once it completes
successfully, to reclaim space (e.g. delete the working directory of an earlier
step consumed as input).

- engine/WorkflowEngine: after a step succeeds, if the step param deleteOnSuccess
  is set, resolve it against the run variables and delete the target (file or
  directory, recursively). Safety: the resolved path must live under a feeds base
  directory and be deeper than a feed root, so it can never remove an entire feed
  or anything outside the feeds area; violations are audited and skipped.
  Best-effort: any failure is audited (DELETE_ON_SUCCESS / _REFUSED / _SKIP /
  _ERROR) and never aborts the run. Applies to the standard step path (not the
  parallel forEach fan-out).
- templates/designer.html: new per-step "On success delete" control (type
  file/directory + path) with autocomplete suggesting the ${dir.<stepId>} of the
  preceding steps. Stored as the step params deleteOnSuccess / deleteOnSuccessType
  (no model/parser/writer change needed).

Compiles (104 classes; +1 is the anonymous file-visitor). Designer passes
node --check.
