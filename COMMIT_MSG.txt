ON HOLD: engine suspend/resume + resume endpoint (5.3 core, Batch F2)

A step can be marked ON HOLD. When the run reaches it the run suspends
(RunStatus.ON_HOLD) without executing the step; a resume (PLAY) executes that step
and continues. Mirrors the existing WAITING_APPROVAL pause/resume infrastructure.

Model: StepDef.onHold (XML onHold="true", round-trips via parser/writer/NodeDto/
toDto like skip); RunStatus.ON_HOLD (non-terminal); WorkflowRun.onHoldStepId and
releasedHold (transient). Designer "Step mode" dropdown becomes
normal / skip / on hold (setStepMode keeps skip & onHold mutually exclusive).

Engine: loop() suspends before executing an onHold step (unless it is the
just-released step), setting ON_HOLD + onHoldStepId, freeing the feed from
runningFeeds, auditing RUN_ON_HOLD; run.currentIndex already tracks position.
resumeHold(feedId,runId,user) sets releasedHold = onHoldStepId, status QUEUED,
re-registers the run and enqueues loop(currentIndex) -- mirrors decide(). ON_HOLD
is added to the active queue snapshot and rank; Stop also aborts an ON_HOLD run.

Endpoint: POST /api/runs/{feedId}/{runId}/resume -> engine.resumeHold.
