# ON HOLD UI: run page + Operations (Batch F3, req 5.3.1/5.3.2/5.3.3)

Completes ON HOLD (builds on F2's engine + /resume endpoint).

## 5.3.1 Run page
- ON_HOLD renders as a BLUE status chip (new `.chip.ON_HOLD` in app.css, same blue
  as RUNNING).
- A dedicated on-hold panel appears when the run is ON_HOLD, showing the held step
  name and the PARTIAL success count "N of TOT steps successful so far". Partial
  outputs are the run.vars already listed in the Variables section.
- A "▶ Continue (resume)" button (and Stop) — resumeRun() POSTs /resume then
  refreshes. ON_HOLD is treated as active so Stop stays available.

## 5.3.2 Operations overview
- ON_HOLD shows as the blue chip in the feed grid (statusChip already uses
  .chip.<STATUS>). bucketFor maps ON_HOLD to a new `onhold` bucket, rendered as an
  "On hold" rollup tile (clickable drill), giving the extra ON HOLD status count.

## 5.3.3 PLAY / CONTINUE
- On the suspended run page: the "Continue (resume)" button (above).
- On the Operations grid: a "▶ Continue" button on rows whose last status is
  ON_HOLD (resumeRun(feedId,runId) -> POST /resume -> reload).
