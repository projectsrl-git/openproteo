# Spec — `csvsql` executor (H2 SQL over CSV → CSV)

> Handoff document for the OpenProteo dev chat. Drop into `.claude/2026-06-19-csvsql-executor-h2.md`.
> Follows `CLAUDE.md` constraints. Implementation is **confirmation-gated, one batch at a time**.
> Docs/commit messages in English; the executor must compile under **Java 8** with **no new compile-time dependency**.

## 1. Goal (user-facing behaviour)

A new internal STEP that runs an arbitrary (possibly complex) SQL query — joins, aggregates,
subqueries, CTEs, window functions — **across several CSV files** and streams the result to a new
CSV, exactly like the other CSV-producing steps.

The user configures, in the designer:

1. **N input CSVs**, each picked with the usual path field (`class="sugin" list="pathsug"`):
   workflow files, step output dirs (`${<stepId>.outputFile}`, `${dir.<stepId>}`), uploaded
   workflow files, shared files, or any `${var}`/literal path — same selection mode as every other step.
   1.1 For each input, a **table name** (the alias the query will use).
2. **One output file** (`csvFile`), same field/mode as `sql`/`split` (`${landingOut}/...`).
3. A **textarea** for the query, with a **Preview** button.
4. The user writes **only the SELECT over the table names**. All H2 plumbing (`CREATE TABLE … AS
   CSVREAD(…)`, the export) is generated by the step and is invisible to the user. The user must be
   **agnostic to H2**.

So for the worked example, the user types only:

```sql
SELECT a.NDG, a.NOME, COUNT(*) AS N_CONTI, SUM(CAST(c.SALDO AS DECIMAL(18,2))) AS TOT
  FROM anagrafica a
  JOIN conti c ON a.NDG = c.NDG
 WHERE c.DATA_APERTURA >= '20150101'
 GROUP BY a.NDG, a.NOME
HAVING SUM(CAST(c.SALDO AS DECIMAL(18,2))) > 0
```

## 2. Design decisions (locked)

- **Everything stays VARCHAR.** Inputs are ingested with `CSVREAD` (which returns every column as
  VARCHAR) and staged as-is. **No typed columns, no indexes, no `dataschema`.** The user casts only
  inside the expressions that actually do arithmetic/date math (`CAST(... AS DECIMAL)`,
  `PARSEDATETIME(...)`), which is plain SQL inside their query — the step adds nothing for typing.
  - Keys stay VARCHAR by design: equi-joins on strings preserve leading zeros (NDG etc.).
  - `yyyyMMdd` dates compare/sort correctly as strings (fixed width), so `WHERE d >= '20150101'`
    works with no conversion.
- **No `CSVWRITE`.** The step does **not** wrap the user query in `CALL CSVWRITE('...','<query>')`
  (that form forces single-quote doubling and re-implements CSV writing). Instead it runs the user
  query with a normal JDBC `Statement.executeQuery(userQuery)` and streams the `ResultSet` through the
  **existing CSV exporter** used by `sql`. Benefits: the user query is passed **verbatim** (no escaping),
  and output gets the house conventions (UTF-8, optional BOM, CRLF, header, **split by rows/MB**, and
  the `csvFile`/`csvFiles`/`csvParts`/`rowCount` vars) for free → full LOOP compatibility.
- **Staging engine:** temporary **file-mode** H2 DB under `${stepDir}` (GB inputs won't fit in
  `mem:`), created at start, **deleted in `finally`**. Inputs staged with `CREATE TABLE <t> AS SELECT *
  FROM CSVREAD(...)` (one full pass per input; bounded memory via MVStore).
- **Pure JDBC, zero compile-time dependency.** The executor uses only `java.sql` + `jdbc:h2:` URLs.
  No `org.h2.*` imports. So the project compiles even with **no H2 on the classpath**; H2 is needed
  only at **runtime**. If the driver is absent the step fails with a clear message (graceful, like ARX).

## 3. XML shape

`<input>` repeating children (modelled on `<replace>`), `<query>` as child text (as `sql` already does),
output via `csvFile` + `csvSplitRows`/`csvSplitMb`/`delimiter` (reused verbatim):

```xml
<step id="join_eor" name="Join EOR + accounts" exec="csvsql"
      csvFile="${landingOut}/report_${runDate}.csv" delimiter=";"
      csvSplitRows="0" csvSplitMb="0">
  <input csv="${extract.outputFile}"      table="anagrafica"/>
  <input csv="${landingOut}/conti.csv"    table="conti"/>
  <query>
    SELECT a.NDG, a.NOME, COUNT(*) AS N_CONTI
      FROM anagrafica a
      JOIN conti c ON a.NDG = c.NDG
     GROUP BY a.NDG, a.NOME
  </query>
</step>
```

`delimiter` (default `;`) is the field separator used for **both** the input `CSVREAD` and the output
CSV (intermediate feeds in a pipeline share one separator — the common case). Per-input separators are
a Batch-2 option.

## 4. Script the step generates (transparent — never shown to the user)

```sql
-- one per <input>, in declaration order
CREATE TABLE anagrafica AS SELECT * FROM CSVREAD('<resolved csv>', NULL, 'fieldSeparator=; charset=UTF-8');
CREATE TABLE conti      AS SELECT * FROM CSVREAD('<resolved csv>', NULL, 'fieldSeparator=; charset=UTF-8');
-- then, NOT as a string literal:
--   ResultSet rs = stmt.executeQuery(<user query, verbatim>);
--   sqlSupport.export(rs, csvFile, delim, header=true, maxRows, maxBytes);  // shared exporter
```

Column names come from each CSV header row (`columnsString = NULL`). All columns VARCHAR.

## 5. Implementation checklist (by file)

> **The four-location rule applies.** Missing the parser whitelist → HTTP 400 with the body hidden by
> the UBS proxy. Register in all four, plus writer/DTO/designer/preview/pom.

**Model**
- `model/def/CsvInput.java` — new tiny class `{ public String csv; public String table; }`
  (mirror `model/def/Replacement.java`).
- `model/def/StepDef.java` — add `public java.util.List<CsvInput> inputs = new ArrayList<>();`.

**Parser — `parser/WorkflowXmlParser.java`**
- Add `"csvsql"` to the **exec whitelist** `Arrays.asList(...)` (and its error message).
- Add `"csvsql".equals(ik)` to the **`internal` boolean**.
- Parse repeating `<input>` children into `s.inputs` (mirror the `<replace>` loop):
  `for (Element in : directChildren(el, "input")) { CsvInput ci = new CsvInput();
   ci.csv = in.getAttribute("csv"); ci.table = in.getAttribute("table"); s.inputs.add(ci); }`.
  (`query`, `csvFile`, `delimiter`, `csvSplitRows`, `csvSplitMb` are already parsed.)

**Engine — `engine/WorkflowEngine.java`**
- Add `e.equals("csvsql")` to `internalKind(String exec)` (line ~583).

**Dispatch + executor — `engine/InternalSteps.java`**
- In `run(...)`: `else if ("csvsql".equals(kind)) { runCsvSql(step, resolvedParams, vars, res, line); }`.
- New `runCsvSql(...)` (signature like `runSql`). Algorithm:
  1. Resolve `csvFile`, `delimiter` (default `;`), each `input.csv`/`input.table` via `VarResolver.resolve`.
  2. Validate: ≥1 input; every input has csv + table; table names match `[A-Za-z_][A-Za-z0-9_]*`
     and are unique; `step.query` non-blank; `csvFile` non-blank. On failure → `line.accept(...)`,
     `res.exitCode = 2`, return.
  3. Temp DB dir: `<resolved ${stepDir}>/_h2`; `mkdirs`. URL
     `jdbc:h2:file:<dir>/q_<runId>_<stepId>;AUTO_SERVER=FALSE` (keep options minimal in Batch 1).
  4. `try { Class.forName("org.h2.Driver"); } catch (ClassNotFoundException e) { line.accept("csvsql:
     H2 driver not on classpath — add com.h2database:h2 (see h2/README_H2.md)"); res.exitCode = 2;
     return; }`
  5. Open connection (pure `DriverManager`). For each input run
     `CREATE TABLE <table> AS SELECT * FROM CSVREAD(?, NULL, 'fieldSeparator=<delim> charset=UTF-8')`
     — pass the path as a parameter where possible; log each as `staged <table> (<n> rows)`.
  6. `Statement st = conn.createStatement(); ResultSet rs = st.executeQuery(query)` — **query verbatim**.
  7. Stream `rs` to `csvFile` through the **shared exporter** (see refactor below), honouring
     `csvSplitRows`/`csvSplitMb`. Publish `##VAR rowCount/csvParts/csvFile/csvFiles` exactly like `runSql`.
  8. `finally`: close conn, then delete the temp DB files (`*.mv.db`, `*.trace.db`) and the `_h2` dir.

- Extract the ResultSet→CSV(+split) core of `exportCsv(...)` into a reusable method, e.g.

## 6. Validation

- Client `clientValidate`: ≥1 input row; each row has csv + table; table names are valid identifiers and
  unique; query non-empty; `csvFile` non-empty.
- Server `runCsvSql` repeats the same checks (designer is not the only entry point — direct XML edit).

## 7. Known gotchas (document in README.md)

- **BOM on inputs.** `sql`/`split` can write a UTF-8 BOM. `CSVREAD` may fold the BOM into the **first
  header cell**, so `${query}` referencing that first column fails. Batch 1: produce csvsql inputs
  **without BOM**. Batch 2: strip a leading BOM on ingest.
- **Empty fields → NULL** (H2 `CSVREAD` default). Use `COALESCE(col,'')` if `''` is required.
- **Disk + time.** Staging copies each input into the temp DB (≈ input size). Ensure `${stepDir}`'s
  volume has room; for very large joins, pre-filter/project upstream (filter early) before this step.

## 8. Outputs / LOOP compatibility

`##VAR rowCount`, `csvFile` (first part), `csvFiles` (delimited list), `csvParts` — identical to `sql`
and `split`, so a downstream `<loop over="${csvFiles}">` works unchanged.

## 9. Out of scope (Batch 2+)

- Per-input field separator / charset / BOM-strip.
- Optional `mem:` mode for small inputs (faster, no disk).
- Opt-in indexes on named join keys for big⋈big joins (one `CREATE INDEX` from a UI field).
- Column-name suggestions per input (read the header) in the designer.
- Optional typed staging driven by `dataschema` (kept off by default per the all-VARCHAR decision).
